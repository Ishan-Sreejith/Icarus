## CRITICAL ISSUE: FUNCTION RETURN VALUES

### Problem Description
All functions return garbage values instead of the correct return value.

Example:
```
fn add_one: x {
    return x + 1
}
var result: add_one: 5
say: result  // Prints garbage instead of 6
```

### Root Cause (Hypothesis)
- Function return values go into x0 register
- Caller expects result in x0
- Issue likely in: Stack frame setup, register preservation, or return instruction
- Affects: ALL function calls

### Impact
- CRITICAL: Blocks any meaningful function usage
- Workaround: Use global code only
- Solution: Fix stack frame and register management

### Fix Needed
1. Verify prologue/epilogue code
2. Check return register (x0) management
3. Verify caller preserves x0 after call
4. Debug stack frame alignment (must be 16-byte aligned)

## CODE CHANGES THIS SESSION

### File: src/jit/compiler.rs

#### Change 1: Added is_map, is_list, is_string builtin handlers

Location: Lines 467-484
```rust
} else if func == "is_map" {
    // Return 1 if argument is a map, 0 otherwise
    // For now, return 0 (not a map) since we don't have map support yet
    if let Some(d) = dest {
        let r = self.regmap.alloc(d)?;
        self.locals.insert(d.clone());
        emit.emit_mov_imm(r, 0); // Maps not supported yet
    }
} else if func == "is_list" {
    // Return 1 if argument is a list, 0 otherwise
    if let Some(d) = dest {
        let r = self.regmap.alloc(d)?;
        self.locals.insert(d.clone());
        emit.emit_mov_imm(r, 0); // Lists not supported yet
    }
} else if func == "is_string" {
    // Return 1 if argument is a string, 0 otherwise
    if let Some(d) = dest {
        let r = self.regmap.alloc(d)?;
        self.locals.insert(d.clone());
        emit.emit_mov_imm(r, 0); // Strings not supported yet
    }
}
```

This pragmatic solution allows type-checking functions to be called without
crashing, returning 0 (false) for all checks since full type support isn't
ready. Once types are fully implemented, these can be updated.
